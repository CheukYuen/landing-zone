## 1. 整体设计思路

### 1.1 核心矛盾

设计本数据库需要同时满足三个相互冲突的需求：

| 需求 | 描述 | 对数据库的要求 |
|------|------|--------------|
| **指标动态变动** | 采购阶段指标频繁增减，稳定后固定 | 加指标不改表结构 |
| **协方差矩阵计算** | 需要"行=日期、列=资产"的宽格式矩阵 | 查询结果直接可用 |
| **LLM 月报生成** | LLM 一次 prompt 读取当月全量数据 | 一行 = 一个月所有指标 |

这三个需求在数据库层面天然冲突，**没有一种表结构能同时满足全部**。解决方案是**分层设计**：不同层解决不同问题。

### 1.2 设计原则

**① 按频率分表，不混存**  
月频宏观数据、日频行情数据、事件驱动数据结构完全不同，混在一张表里会造成大量空值和查询复杂度爆炸。

**② 原始数据层用窄表，聚合层用宽表**  
原始数据层的窄表解决"写入灵活"的问题；聚合层的宽表解决"读取高效"的问题。两层通过每月一次的聚合脚本同步。

**③ 矛盾上移到 Python 层**  
数据库无法同时做到"灵活写入"和"直接计算"，把格式转换（pivot）交给 Python 封装，对业务调用方透明。

**④ indicator_catalog 作为唯一真相来源**  
所有指标的元信息（含义、单位、来源）集中在目录表，数据库通过外键约束保证一致性，LLM 通过联表查询获取语义上下文。**内部合成特征同样在 catalog 中登记**（`data_source='INTERNAL_CALC'`），保证所有可引用指标都有元数据。

**⑤ 原始文件保留在文件系统，不入库**  
外部采购文件（xlsx、CSV 等）原样保存在本地文件夹，**不新建数据库表存储**。ETL 脚本直接读取文件清洗后写入 Layer 1。原因：文件本身就是最好的归档格式，数据库表反而增加维护成本。约定文件命名规范即可实现溯源。

**⑥ 日频原始保留策略**（v1.3 新增）  
对于 DR007、YIELD_30Y 等明确要求"保留日频原始 + 输出月均"的指标，日频原始数据写入 `macro_daily`（新增表），月均聚合值写入 `macro_monthly` 和宽表。确保审计/复算可溯源至日频粒度。

## 2. 核心概念：窄表 vs 宽表

### 2.1 窄表（Narrow Table）

**定义**：每一行只存「一个指标 + 一个时间点 + 一个值」。行数多，列数少（通常 4~6 列）。

**示例** — `macro_monthly` 窄表：

| indicator_code | stat_month | value | data_version |
|----------------|------------|-------|--------------|
| CPI_YOY        | 2026-01    | 0.5   | 1            |
| PPI_YOY        | 2026-01    | -2.3  | 1            |
| M2_YOY         | 2026-01    | 7.3   | 1            |
| CPI_YOY        | 2026-02    | 0.6   | 1            |
| **YIELD_30Y_AVG** | **2026-02** | **2.85** | **1** |

> ✅ 新增指标直接 `INSERT`，**无需修改表结构**

**优点**：
- 指标随时增减，零改表成本
- 支持 `data_version` 字段追踪数据修订（统计局会修订历史数据）
- 单表存全部宏观指标，维护简单

**缺点**：
- 计算协方差矩阵前必须做 pivot 转置
- 查询"某月所有指标"需要聚合操作，SQL 较复杂

---

### 2.2 宽表（Wide Table）

**定义**：每一行存「一个时间点的所有指标值」。行数少，列数多（每个指标一列）。

**示例** — `macro_snapshot_monthly` 宽表：

| snap_month | cpi_yoy | ppi_yoy | m2_yoy | yield_30y_avg | t5yie_avg | pboc_rate_event |
|------------|---------|---------|--------|---------------|-----------|-----------------|
| 2026-01    | 0.5     | -2.3    | 7.3    | 2.88          | 2.35      | 无调整          |
| 2026-02    | 0.6     | -2.1    | 7.5    | 2.85          | 2.31      | LPR降息10bp     |

> ✅ 一行 = 一个月的完整数据快照，LLM 一次 `SELECT` 直接渲染 Markdown

**优点**：
- 查询极快，无需聚合
- LLM prompt 友好，直接映射为 Markdown 表格
- 协方差计算可直接 `df.cov()`

**缺点**：
- 新增指标需要 `ALTER TABLE` 加列（生产环境有风险）
- 不适合频繁变动的采购阶段

---

### 2.3 分工总结

```
写入 / 维护  →  窄表（macro_monthly, macro_daily, market_daily, quant_daily）
                每月 ETL 脚本直接 INSERT，指标变动零成本

读取 / 计算  →  宽表（macro_snapshot_monthly）
                每月聚合一次，LLM / 报告直接消费

格式转换     →  Python 层封装 pivot 函数
                协方差矩阵计算对调用方透明
```

## 4. 各表 DDL

### 4.1 Layer 0 — 指标目录

```sql
-- 作用：① 防脏数据写入（外键约束）
--       ② 给 LLM 提供语义上下文（联表查询）
--       ③ 管理指标生命周期（is_active 字段）
--       ④ 登记内部合成特征（data_source='INTERNAL_CALC'）
-- ============================================================
CREATE TABLE indicator_catalog (
    indicator_code   VARCHAR(60)   NOT NULL  COMMENT '指标代码, 如 CPI_YOY',
    indicator_name   VARCHAR(120)  NOT NULL  COMMENT '中文名称',
    category         ENUM(
                       'macro_core',       -- 宏观核心（16个）
                       'macro_secondary',  -- 宏观非核心（14个）
                       'market',           -- 行情数据（16个）
                       'quant',            -- 量化建模（38个）
                       'news_policy',      -- 新闻政策（12个）
                       'internal_derived'  -- 内部合成特征（非采购）
                     )             NOT NULL,
    data_source      VARCHAR(80)   NOT NULL  COMMENT '数据来源, 如 NBS/PBOC/Wind/INTERNAL_CALC',
    frequency        ENUM('daily','weekly','monthly','quarterly','event')
                                   NOT NULL,
    value_type       ENUM('yoy','mom','qoq','qoq_sa','level',
                          'cumulative_yoy','index','rate','spread')
                                   NOT NULL  COMMENT '调整方式',
    unit             VARCHAR(40)   NULL      COMMENT '单位, 如 %/点/USD/bp',
    history_window   VARCHAR(40)   NULL      COMMENT '近10年/近5年 等',
    target_table     VARCHAR(60)   NOT NULL  COMMENT '实际存储的 Layer 1 表名',
    is_active        TINYINT(1)    DEFAULT 1 COMMENT '0=已停用，历史数据保留',
    notes            TEXT          NULL      COMMENT '业务含义/公式说明（供LLM读取）',
    updated_at       TIMESTAMP     DEFAULT CURRENT_TIMESTAMP
                                   ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (indicator_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='指标目录/数据字典 — Layer 0';
```

**v1.3 变更说明**：
- `category` ENUM 新增 `'internal_derived'`，用于登记内部合成特征
- `value_type` ENUM 新增 `'qoq_sa'`（季调环比，用于 `REAL_GDP_QOQ`）和 `'spread'`（利差类）
- `macro_core` 注释从14个更新为16个
- `market` 注释从15个更新为16个

### 4.2 Layer 1a — 宏观月频时序

```sql
-- ============================================================
-- macro_monthly: 宏观月频时序（窄表）
-- 覆盖：宏观核心16个（月均聚合值） + 宏观非核心14个 = 30个指标
-- 设计要点：data_version 支持统计局数据修订追踪
-- 注意：DR007/YIELD_30Y/T5YIE/CREDIT_SPREAD等日频指标的
--       月均聚合值也写入本表，日频原始保留在 macro_daily
-- ============================================================
CREATE TABLE macro_monthly (
    id               BIGINT        NOT NULL AUTO_INCREMENT,
    indicator_code   VARCHAR(60)   NOT NULL  COMMENT 'FK → indicator_catalog',
    stat_month       CHAR(7)       NOT NULL  COMMENT '统计月份 YYYY-MM',
    value            DECIMAL(16,6) NULL      COMMENT '指标值（NULL=本月未发布）',
    is_preliminary   TINYINT(1)    DEFAULT 0 COMMENT '1=初值，可能被后续修订',
    data_version     TINYINT       DEFAULT 1 COMMENT '修订版本号，查询取MAX',
    source_tag       VARCHAR(40)   NULL      COMMENT '来源标识, 如 NBS_OFFICIAL / AGG_FROM_DAILY',
    created_at       TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    updated_at       TIMESTAMP     DEFAULT CURRENT_TIMESTAMP
                                   ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_code_month_ver (indicator_code, stat_month, data_version),
    INDEX idx_stat_month         (stat_month),
    INDEX idx_indicator_code     (indicator_code),
    CONSTRAINT fk_macro_catalog
        FOREIGN KEY (indicator_code) REFERENCES indicator_catalog(indicator_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='宏观月频时序数据 — Layer 1a（含日频指标的月均聚合值）';
```

> **修订处理逻辑**：统计局发布修订数据时，**不覆盖原行**，追加 `data_version=2` 的新行。查询时 `WHERE data_version = (SELECT MAX(data_version) FROM ... WHERE ...)` 取最新版本。

> **日频→月均聚合**：DR007、YIELD_10Y、YIELD_30Y、T5YIE、CREDIT_SPREAD_AA、CREDIT_SPREAD_AA_AAA、FX_CNY_MID 的月均值在 ETL 阶段由 `macro_daily` 聚合后写入，`source_tag='AGG_FROM_DAILY'`。

### 4.2b Layer 1a-daily — 宏观日频时序（v1.3 新增）

```sql
-- ============================================================
-- macro_daily: 宏观日频时序（窄表）
-- 覆盖：需要保留日频原始的宏观指标
--   - DR007（央行流动性锚定）
--   - YIELD_10Y（10年国债日频）
--   - YIELD_30Y（30年国债日频，全天候P0）
--   - T5YIE（美国5年通胀预期日频）
--   - CREDIT_SPREAD_AA（AA信用利差日频）
--   - CREDIT_SPREAD_AA_AAA（AA-AAA等级利差日频）
--   - FX_CNY_MID（人民币中间价日频）
-- 设计要点：
--   ① 日频原始保留，支持审计/复算溯源
--   ② 月均聚合由 ETL 写入 macro_monthly，source_tag='AGG_FROM_DAILY'
--   ③ 与 market_daily 区分：本表存宏观利率/汇率类，
--      market_daily 存股指/波动率/资金流/大宗
-- ============================================================
CREATE TABLE macro_daily (
    id               BIGINT        NOT NULL AUTO_INCREMENT,
    indicator_code   VARCHAR(60)   NOT NULL  COMMENT 'FK → indicator_catalog',
    trade_date       DATE          NOT NULL  COMMENT '交易日/发布日',
    value            DECIMAL(16,6) NULL      COMMENT '指标值',
    created_at       TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_code_date (indicator_code, trade_date),
    INDEX idx_trade_date      (trade_date),
    INDEX idx_indicator_code  (indicator_code),
    CONSTRAINT fk_macro_daily_catalog
        FOREIGN KEY (indicator_code) REFERENCES indicator_catalog(indicator_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='宏观日频时序数据 — Layer 1a-daily（v1.3新增，保留日频原始）';
```

### 4.3 Layer 1b — 行情日频时序

```sql
-- ============================================================
-- market_daily: 行情日频时序（窄表）
-- 覆盖：16个 ticker（股指/波动率/资金流向/大宗商品）
-- v1.3 变更：新增 IRON_ORE_DCE、SOUTHBOUND_NET，共16个

CREATE TABLE market_daily (
    id               BIGINT        NOT NULL AUTO_INCREMENT,
    ticker           VARCHAR(30)   NOT NULL  COMMENT '如 SPX / 000001.SH / VIX',
    trade_date       DATE          NOT NULL  COMMENT '交易日',
    close_price      DECIMAL(18,6) NULL      COMMENT '收盘价/指数点位',

    net_flow         DECIMAL(18,4) NULL      COMMENT '净流入额，仅资金流向类（亿元）',
    created_at       TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_ticker_date (ticker, trade_date),
    INDEX idx_trade_date      (trade_date),
    INDEX idx_ticker          (ticker)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='行情日频时序数据 — Layer 1b（16 ticker）';
```

```
├─ Step 3: 数据质量校验
│   ├── 值域检查：3σ 以外异常值标记人工复核

├─ Step 4: 聚合写入 Layer 2 宽表
│   └── CALL proc_build_monthly_snapshot('YYYY-MM')

│
└─ Step 5: 触发下游任务
    ├── LLM 月报生成
    └── 协方差矩阵更新
```

### 场景 2：计算协方差矩阵（含30Y长债）

**需求**：读取近 10 年量化建模指标（含30Y），计算年化协方差矩阵。

**Step 1 — SQL 读取窄表**：

```sql
SELECT ticker, trade_date, close_price, yield_rate
FROM quant_daily
WHERE trade_date >= '2016-01-01'
  AND trade_date <= '2026-01-01'
  AND ticker IN ('510300', '518880', 'NHCI', 'SHIBOR_1W', 'CN_BOND_10Y', 'CN_BOND_30Y')
ORDER BY trade_date, ticker;
```

**Step 2 — Python 封装（对业务层透明）**：

```python
import pandas as pd
import numpy as np

def get_cov_matrix(conn, tickers: list[str], start: str, end: str) -> pd.DataFrame:
    """
    从 quant_daily 窄表读取，pivot 为宽格式，计算年化协方差矩阵
    调用方无需感知底层是窄表还是宽表
    """
    placeholders = ','.join(['%s'] * len(tickers))
    sql = f"""
        SELECT ticker, trade_date, close_price
        FROM quant_daily
        WHERE ticker IN ({placeholders})
          AND trade_date BETWEEN %s AND %s
        ORDER BY trade_date
    """
    df = pd.read_sql(sql, conn, params=[*tickers, start, end])

    # 窄表 → 宽格式（行=日期，列=ticker）
    wide = df.pivot(index='trade_date', columns='ticker', values='close_price')

    # 计算日度收益率，去掉 NaN
    returns = wide.pct_change().dropna()

    # 年化协方差矩阵（× 252 个交易日）
    cov_matrix = returns.cov() * 252

    return cov_matrix

# 调用示例（v1.3: 含30Y）
tickers = ['510300', '518880', 'NHCI', 'SHIBOR_1W', 'CN_BOND_10Y', 'CN_BOND_30Y']
cov = get_cov_matrix(conn, tickers, '2016-01-01', '2026-01-01')
print(cov.round(4))
```

```sql
SELECT
    d.trade_date,
    d.value AS term_spread_10y_1y
FROM derived_daily d
WHERE d.indicator_code = 'TERM_SPREAD_10Y_1Y'
  AND d.trade_date >= '2024-03-01'
```

**联合30Y长端利差**：

```sql
SELECT
    a.trade_date,
    a.value AS spread_10y_1y,
    b.value AS spread_30y_10y
FROM derived_daily a
LEFT JOIN derived_daily b
    ON a.trade_date = b.trade_date
    AND b.indicator_code = 'TERM_SPREAD_30Y_10Y'
WHERE a.indicator_code = 'TERM_SPREAD_10Y_1Y'
  AND a.trade_date >= '2024-03-01'
ORDER BY a.trade_date;
```

### `quant_daily` 迁移宽表的判断标准

当以下条件**同时满足**时，执行迁移：

- [ ] 六资产（含 CN_BOND_30Y）+ Benchmarks 的 ticker 集合连续 2 个月无变动
- [ ] 协方差矩阵计算频率 > 每日 1 次（pivot 性能开始成为瓶颈）
- [ ] 数据回测窗口已扩展至 10 年完整历史

**迁移 SQL**（一次性执行）：

```sql
-- 窄表数据透视迁移到宽表
INSERT INTO asset_returns_daily (trade_date, price_510300, price_518880,
    price_nhci, rate_shibor_1w, yield_cgb_10y, yield_cgb_30y)
SELECT
    trade_date,
    MAX(CASE WHEN ticker='510300'      THEN close_price END) AS price_510300,
    MAX(CASE WHEN ticker='518880'      THEN close_price END) AS price_518880,
    MAX(CASE WHEN ticker='NHCI'        THEN close_price END) AS price_nhci,
    MAX(CASE WHEN ticker='SHIBOR_1W'   THEN yield_rate  END) AS rate_shibor_1w,
    MAX(CASE WHEN ticker='CN_BOND_10Y' THEN yield_rate  END) AS yield_cgb_10y,
    MAX(CASE WHEN ticker='CN_BOND_30Y' THEN yield_rate  END) AS yield_cgb_30y
FROM quant_daily
GROUP BY trade_date
ORDER BY trade_date;
```
